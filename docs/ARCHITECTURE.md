# Архитектура (накратко)

```
   Flutter (Android / iOS / Desktop)
            │
            ▼
      Isar  (локална база, offline-first)   ←── UI чете САМО оттук
            │
       SyncService
        │        ▲
   push │        │ pull (since=cursor)
        ▼        │
        FastAPI  (sync engine)
            │
        PostgreSQL
```

## Принципът на sync-а
Всеки запис (`Memory`) има:
- `uid` — глобален UUID, генериран от клиента. Затова два телефона никога не правят един и същи id.
- `updated_at` — последна промяна. Сливането е **last-write-wins**: по-новото време печели.
- `rev` — сървърен монотонен номер. Клиентът пита „дай ми всичко с `rev` по-голям от моя курсор".
- `deleted` — меко изтриване. Изтритото също се синхронизира, за да изчезне навсякъде. Нищо не се трие физически.

UI пише в Isar и веднага вижда резултата. SyncService работи отделно: качва непратеното, сваля чуждото. Ако няма мрежа — нищо не блокира, опитва пак след 15 секунди.

## Какво е тествано и какво е скелет
- **server/** — реален и тестван: push/pull, last-write-wins (старите промени се отхвърлят), инкрементален pull, меко изтриване. Работи на SQLite за разработка и на PostgreSQL в Docker.
- **app/** (Flutter) — правилно структуриран скелет. Компилира се при теб: `flutter pub get && dart run build_runner build && flutter run`. Не е компилиран в средата, в която е генериран.
- **Уеб клиент** — вграден в backend-а (`server/app/web`). Отваряш `http://сървър:8000` и ползваш веднага, без Flutter.

## Какво НЕ е тук (нарочно)
Voice, Whisper, локален LLM, AI тагове, Mind Box, Guardian, графики. Те влизат само след като това се ползва. Правилото остава: нова функция влиза само ако намалява броя решения.
